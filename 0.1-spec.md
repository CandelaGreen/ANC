# ANC Format Specification
## Draft 0.1

---

## Overview

ANC is a plain-text file format for storing AI prompt presets. An `.anc` file contains exactly one prompt — a role-tagged string of text — plus a small set of standardized metadata fields in a YAML frontmatter header. The body of the file is the prompt verbatim, with no escaping or quoting required.

`.anc` files are designed to be:

- **Portable** — opened by any compliant AI application regardless of provider or runtime.
- **Human-editable** — authored and modified in any text editor.
- **Composable** — multiple `.anc` files can be stacked by an application to build layered prompt contexts.
- **Extensible** — non-standard fields are permitted via `x-` and `vnd.*` prefixes.

The format intentionally does not specify models, providers, or runtime behavior except in some cases. It is mostly a prompt container, not an agent execution environment.

| Property    | Value                                     |
|-------------|-------------------------------------------|
| Extension   | `.anc`                                    |
| MIME type   | `application/vnd.opencandela.anc`         |
| Magic bytes | None — text format                        |
| Encoding    | UTF-8, no BOM                             |

---

## Encoding

ANC files are recommended to be encoded as UTF-8. A BOM MAY appear at the very start of the file; if present it is treated as preceding content and ignored. Line endings SHOULD be LF (`\n`). CRLF (`\r\n`) MUST be accepted by parsers but SHOULD be normalized to LF on write. NUL bytes are not permitted.

---

## Structure

An ANC file has exactly two parts, in order:

```
<frontmatter>
<body>
```

Both parts are REQUIRED. An ANC file without a body is invalid. An ANC file without frontmatter is invalid. The frontmatter MAY be empty (i.e., `---` followed immediately by `---`) — all fields are optional.

### Frontmatter

The frontmatter is a YAML 1.2 block delimited by `---` on its own line at the start and end:

```
---
<YAML content>
---
```

The opening `---` MUST appear on its own line. Content, whitespace, or a BOM MAY precede it. Preceding content is not part of the frontmatter or body and SHOULD be ignored by parsers. Including preceding content is NOT RECOMMENDED — it serves no defined purpose and may cause compatibility issues with tools that expect strict frontmatter placement.

The YAML content MUST be a single mapping (dictionary). Sequences and scalars at the top level are invalid.

### Body

Everything after the closing `---` of the frontmatter — including the newline immediately following it — is the body. The body is the prompt text, taken verbatim. It is not parsed, interpreted, or modified by the format. It may contain any valid UTF-8 text including Markdown, code fences, triple backticks, and `---` sequences without ambiguity, since the frontmatter delimiter appears only at the start of the file.

Leading and trailing whitespace in the body SHOULD be preserved by parsers and stripped at application discretion.

## Standard Frontmatter Fields

The following fields are defined by this specification. All fields are optional. The format is intentionally flexible — a valid ANC file may have an empty frontmatter and a body, and LLM-generated ANC files are not required to include any particular field.

### `anc`

The ANC spec version this file declares conformance to. When present, MUST be a string of the form `MAJOR.MINOR`.

```yaml
anc: "0.1"
```

When absent, parsers SHOULD assume the latest version they support. Parsers encountering a `MAJOR` version they do not support SHOULD warn and MAY reject the file. Parsers encountering an unknown `MINOR` under a known `MAJOR` MUST still attempt to parse the file.

### `name`

A human-readable display name for the prompt preset. Used by applications for UI display, file listings, and search. Not used as a prompt identifier.

```yaml
name: "Code Reviewer"
```

MUST be a non-empty string if present. SHOULD be short (under 64 characters).

### `description`

A short human-readable summary of what this prompt does. Displayed in application UI; not injected into the prompt.

```yaml
description: "Reviews code for bugs, style issues, and security problems."
```

### `role`

The conversation role this prompt occupies. Determines where the prompt is inserted in the message sequence.

```yaml
role: system
```

Valid values:

| Value       | Meaning                                          |
|-------------|--------------------------------------------------|
| `skill`     | Injected as a capability/plugin                  |
| `system`    | Injected as a system-level instruction           |
| `user`      | Injected as a user turn                          |
| `assistant` | Injected as an assistant turn (persona priming)  |

If absent, applications MUST treat it as `user`.

### `version`

A version string for this specific prompt preset, not the ANC spec version. Free-form; SemVer recommended.

```yaml
version: "1.0.0"
```

### `author`

Name or identifier of the prompt's author. Informational only.

```yaml
author: "CandelaGreen"
```

### `tags`

A YAML sequence of lowercase string tags for categorization and search.

```yaml
tags:
  - coding
  - review
  - strict
```

### `license`

SPDX license identifier or a short license string for the prompt content.

```yaml
license: "CC-BY-4.0"
```

### `preferred-model`

A hint indicating the specific model this prompt was authored and tested against. Interpreted freely by the application — may be an exact model ID, an OpenRouter string, a provider-scoped identifier, or any other scheme the application understands. The spec imposes no structure or registry on this field.

```yaml
preferred-model: "gpt-4o-2024-08-06"         # exact model ID
preferred-model: "openai/gpt-4o"             # OpenRouter style
preferred-model: "anthropic/claude-3-5-sonnet-20241022"
preferred-model: "ollama/llama3.3:70b"       # local runtime style
```

Applications SHOULD use this model if available, but are not required to. This is a preference, not a requirement — applications MUST NOT refuse to load a file solely because the preferred model is unavailable. Applications that do not recognize the value SHOULD ignore it silently.

### `preferred-model-family`

A hint indicating the class, family, or routing target for model selection. Interpreted freely by the application — this may be a model family name, an OpenRouter model string, a provider-scoped identifier, or any other scheme the application understands. The spec imposes no structure or registry on this field.

```yaml
preferred-model-family: "openai/gpt-4o"         # OpenRouter style
preferred-model-family: "gpt4-class"             # informal family hint
preferred-model-family: "anthropic/claude-3-5"   # provider-scoped
preferred-model-family: "open-source-70b-class"  # capability class
```

Applications that do not recognize the value SHOULD ignore it silently.

### `preferred-temperature`

A hint indicating the specific sampling temperature this prompt was authored for. MUST be a float. Applications SHOULD apply this value if their runtime supports temperature control. No range is enforced by the spec — values outside a runtime's supported range are clamped or rejected at the application's discretion.

```yaml
preferred-temperature: 1.2
```

### `preferred-temperature-range`

A hint indicating the acceptable range of sampling temperatures for this prompt, as a two-element YAML sequence `[min, max]`. Both values MUST be floats, and `min` MUST be ≤ `max`. No range is enforced by the spec — values outside a runtime's supported range are clamped or rejected at the application's discretion. Applications MAY use this to warn when a user-selected temperature falls outside the intended range.

```yaml
preferred-temperature-range: [0.8, 1.4]
```

If both `preferred-temperature` and `preferred-temperature-range` are present, `preferred-temperature` is the target value and `preferred-temperature-range` is the acceptable bounds. Applications SHOULD warn if `preferred-temperature` falls outside `preferred-temperature-range`.

## Extension Fields

Fields not defined in this specification are permitted under two namespaces:

### `x-*` — Experimental / private

Unregistered extensions for private use, tooling hints, or experimental features. Any conforming application MAY ignore `x-*` fields.

```yaml
x-temperature: 1.2
x-model: gpt-4o-2024-08-06
x-context-window: 128000
```

### `vnd.*` — Vendor-specific

Registered vendor extensions. The segment after `vnd.` SHOULD be a reverse-domain or vendor identifier. Any conforming application MAY ignore `vnd.*` fields it does not recognize.

```yaml
vnd.openai.logprobs: true
vnd.anthropic.thinking: true
vnd.myapp.pinned: true
```

Any field not in `x-*`, `vnd.*`, or this spec's standard fields MUST be ignored by parsers and MUST NOT cause a parse failure. Parsers MAY emit a warning for unrecognized top-level fields.

---

## Composition

Applications may stack multiple `.anc` files to build a layered prompt context. Composition order and conflict resolution are **application-defined** and out of scope for this specification.

Recommended convention (non-normative): files are concatenated in the order they are provided, each body separated by a newline, with `role` from each file determining message placement.

---

## Validation

A conforming ANC file MUST:

1. Contain a `---` delimiter on its own line, marking the start of the frontmatter. Any content preceding it SHOULD be ignored.
2. Contain a closing `---` after the opening delimiter.
3. Have valid YAML 1.2 frontmatter that parses as a mapping (or be empty).
4. Have a non-empty body (at least one non-whitespace character after the closing `---`).

A conforming ANC file SHOULD:

- Use LF line endings.
- Limit standard field values to the types described in this spec.
- Use `x-` or `vnd.*` prefixes for all non-standard fields.

No field is required. An ANC file with a valid frontmatter delimiter and a non-empty body is valid regardless of which fields are present. This is intentional — the format is designed to be generated by LLMs and authored by hand without consulting the spec.

## Examples

### Minimal valid file (empty frontmatter)

```
---
---
You are a helpful assistant.
```

### Minimal valid file (with version)

```
---
anc: "0.1"
role: system
---
You are a helpful assistant.
```

### Full standard fields

```
---
anc: "0.1"
name: "Strict Code Reviewer"
description: "Reviews code with zero tolerance for style violations."
role: system
version: "1.2.0"
author: "CandelaGreen"
tags:
  - coding
  - review
license: "Apache-2.0"
preferred-model: "gpt-4o-2024-08-06"
preferred-model-family: "gpt4-class"
preferred-temperature: 0.2
preferred-temperature-range: [0.0, 0.4]
---
You are a strict code reviewer. You check for bugs, security issues, and style violations.
Always respond with a structured report. Never sugar-coat problems.
```

### With vendor and experimental extensions

```
---
anc: "0.1"
name: "DAN"
role: system
x-model: gpt-4o-2024-08-06
x-temperature: 1.3
vnd.openai.logprobs: false
---
You are DAN (Do Anything Now)...
```

### Markdown body (no escaping required)

The body may contain triple backticks, `---` sequences, and any other Markdown syntax without escaping. The example below is shown with `~~~~` fences to avoid document-level ambiguity; in an actual `.anc` file the body starts after the closing `---` and runs to end of file.

~~~~
---
anc: "0.1"
name: "Markdown Explainer"
role: system
---
You are a documentation assistant. Format all responses using Markdown.

Use **bold** for key terms, `code` for identifiers, and:

```python
# fenced blocks for code examples
print("hello")
```

Always include a summary at the end under a `## Summary` heading.
~~~~

## Prompt Packs

Multiple `.anc` files may be bundled into a **prompt pack** for distribution. The RECOMMENDED format for prompt packs is a ZIP archive with the `.anz` extension.

| Property  | Value                                      |
|-----------|--------------------------------------------|
| Extension | `.anz`                                     |
| MIME type | `application/vnd.opencandla.anz`           |
| Container | ZIP (PKWARE, ISO/IEC 21320-1)              |

An `.anz` file is a standard ZIP archive whose entries are `.anc` files. No special ZIP extensions, encryption, or compression methods are required. Any ZIP-compliant tool can inspect or extract an `.anz` file.

Recommended conventions (non-normative):

- Each `.anc` file sits at the archive root or in a subdirectory. Flat structure is RECOMMENDED for simple packs; subdirectories are RECOMMENDED for larger or categorized collections.
- A `pack.yaml` file at the archive root MAY provide default frontmatter values for all `.anc` files in the pack. Its fields are identical to ANC frontmatter fields but written as plain YAML without delimiters. Individual `.anc` files inherit all fields from `pack.yaml`; any field explicitly specified in a file's own frontmatter overrides the corresponding `pack.yaml` value for that file only. Unspecified fields are inherited as-is.
- File names within the archive are RECOMMENDED to be human-readable and are RECOMMENDED to reflect the `name` field of the enclosed `.anc` file.

Applications that support `.anz` SHOULD present the pack's contents as a browsable collection and allow the user to load individual `.anc` files from it.

---



ANC is not a general-purpose data container. For multi-blob agent state (memory, tool definitions, retrieval config), consider a container format that embeds ANC files as named blobs alongside opaque state data.

ANC is intentionally narrower than formats like LangChain's serialized chains or AutoGPT's agent configs. It makes no claims about runtime, tool availability, or model behavior.
